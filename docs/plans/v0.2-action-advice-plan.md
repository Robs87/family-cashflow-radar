# 家庭现金流行动建议 v0.2 实施计划

## 1. 目标

v0.2 把系统从“现金流报表”推进到“消费行动建议”。

2026-05-18 需求调整后，BeeCount Cloud 成为流水记录层。本计划中“统一新增交易服务”和“Web 手动记录入口”降级为过渡能力；后续主线改为 BeeCount Cloud 数据源适配和分析决策引擎。

最短路径：

```text
BeeCount Cloud 流水同步 -> 标准化现金流语义 -> 固定支出自动化 -> 安全垫计算 -> 首页一句建议 -> 决策模拟器
```

v0.2 的验收标准不是页面更丰富，而是用户打开 Web 首页后能立刻知道：

- 家庭现金流现在安全不安全。
- 未来 30 天要不要避免大额消费。
- 当前适不适合提前还贷。
- 当前最多能拿多少钱去投资而不破坏安全垫。
- 本月哪类可控支出需要压住。

## 1.1 当前状态

截至 2026-05-22：

- BeeCount read API 同步、refresh token 自动刷新和 Keychain 配置页已实现。
- BeeCount 分类映射层和 Web 映射维护入口已实现。
- 固定账单、房贷计划、提前还贷事件、自动记账和编辑入口已实现。
- 首页现金流安全判断和近期建议已实现第一版。
- 决策模拟器已实现第一版，提前还贷已支持现金流压力和利息节省估算。

仍未完成、且会直接影响建议可信度的主线：

- 当前现金余额校准。
- 未来计划事件。
- 计划事件与 BeeCount 实际流水匹配去重。
- 大额消费 / 买车分期成本。
- 投资仓位安全上限。

## 2. 数据语义补充

v0.1 的金额规则继续有效：

- 金额使用整数分。
- `amount_cents` 永远非负。
- 真实方向由 `cashflow_direction` 表达。
- 人工覆盖优先。
- 分类规则可解释。

v0.2 新增记录语义：

```text
stable_income
living_expense
fixed_expense
debt_payment
investment_inflow
investment_outflow
internal_transfer
reimbursable_expense
reimbursement_income
asset_purchase
asset_sale
one_time_income
```

其中 `reimbursable_expense` 和 `reimbursement_income` 不进入家庭生活支出和稳定收入，但会影响短期现金余额。

## 3. 推荐实施顺序

### Task 1：BeeCount Cloud 数据源适配（完成）

目标：把 BeeCount Cloud 作为默认流水来源，只读同步交易到本地分析库。

要求：

- 支持配置 BeeCount Cloud base URL、账本 ID 和只读 token。
- 区分 MCP PAT 和 read API access token：`bcmcp_...` PAT 只能访问 `/api/v1/mcp/*`，普通 `/api/v1/read/*` 同步需要 Web/App access token。
- 优先使用只读能力：MCP read tools、read API 或离线 SQLite 备份。
- 读取交易、账户、分类、标签和预算。
- 写入本地 `raw_transactions`，保留 BeeCount 原始 payload。
- 为 BeeCount 交易生成稳定幂等键，重复同步不得重复计数。
- 不默认向 BeeCount Cloud 写交易。

验收：

```bash
uv run --with pytest pytest tests/test_beecount_import.py tests/test_normalize.py tests/test_monthly_cashflow.py
```

已补齐：

- 明确 BeeCount 交易被修改、删除、恢复时，本地 raw/normalized/monthly 如何处理。
- 同一 BeeCount 交易多版本进入本地库时，月度现金流只能使用最新有效版本。
- 新增删除交易和修改交易的合成 fixture。

### Task 1b：本地新增交易服务降级为补充能力

现有对话式记录和 Web 手动输入可以保留，但只用于：

- BeeCount 尚未接入前的过渡记录。
- 分析事件、现金余额校准、模拟参数。
- 明确不需要回写 BeeCount 的临时记录。

后续如果要把本项目中的新增交易写回 BeeCount Cloud，必须单独设计权限、审计和冲突处理，不能绕过 BeeCount 的记录层。

### Task 2：补齐记录类型和分类规则（持续进行）

目标：让系统能正确处理工作垫付、报销、投资、债务、资产事件。

要求：

- 新增或校准 `reimbursable_expense`。
- 新增或校准 `reimbursement_income`。
- 报销到账不算稳定收入。
- 工作垫付不算生活支出。
- 投资流入不算收入。
- 投资流出不算消费。
- 信用卡还款和内部转账保持 neutral。

验收：

```bash
uv run --with pytest pytest tests/test_classify.py tests/test_monthly_cashflow.py
```

### Task 2b：BeeCount 分类映射层（基本完成，持续扩展）

目标：BeeCount 分类和家庭现金流语义保持一致，并支持 BeeCount 新增分类后继续维护映射。

要求：

- 同步或导入 BeeCount 分类时，自动补充 `beecount_category_mappings`。
- 对无法确定语义的新分类，标记为待映射，不直接进入可信建议。
- Web 端允许用户维护 cashflow_direction、financial_type、一级/二级分类。
- 修改映射后可重新 classify 并刷新首页建议。

验收：

```bash
uv run --with pytest pytest tests/test_beecount_import.py tests/test_classify.py tests/test_web_dashboard.py
```

### Task 3：固定支出和房贷计划（基本完成）

目标：确定性支出自动生成，减少每月手动记录。

要求：

- 房贷模板自动生成还款计划。
- 房贷记录拆分本金和利息。
- 提前还贷事件可保存、可修改、可重算。
- 宽带、电话费、保险等固定账单按模板生成。
- Web 使用清单展示，修改通过单独按钮进入编辑区。

验收：

```bash
uv run --with pytest pytest tests/test_recurring.py tests/test_web_dashboard.py
```

### Task 4：首页建议引擎（第一版完成，待接入现金余额和预测）

目标：Web 首页输出一条核心建议。

建议格式：

```text
当前家庭现金流：观察状态。未来 30 天不建议新增 3 万元以上大额支出；可以继续正常消费，但提前还贷建议等下次稳定收入到账后再评估。
```

要求：

- 计算 safe / watch / tight / danger。
- 计算现金安全垫月数。
- 找出未来 30 天和未来 3 个月的风险。
- 输出建议原因。
- 当 unknown / pending 太多时，降低建议可信度并提示先补分类。

验收：

```bash
uv run --with pytest pytest tests/test_web_dashboard.py
```

### Task 5：提前还贷模拟器（第一版完成）

目标：回答“现在能不能提前还贷”。

输入：

- 还款金额。
- 还款日期。
- 是否重算月供或缩短期限。

输出：

- 执行后现金安全垫。
- 未来最低现金余额月份。
- 节省利息估算。
- 建议执行、暂缓、或降低金额。

### Task 6：当前现金余额校准（已完成）

目标：给安全垫和预测一个真实起点，避免只用“本月基础结余”替代可用现金。

输入：

- 当前可用现金余额。
- 统计口径：家庭现金账户、活期、货币基金等可快速动用资金。
- 校准日期。

输出：

- 当前可用现金。
- 现金可覆盖固定支出和债务还款的月数。
- 后续决策模拟的 opening cash。

已落地：

- `cash_balance_calibrations` 保存校准日期、可用现金、统计口径和备注。
- 首页财务建议区可录入和展示最新校准。
- 财务建议优先使用校准现金计算安全垫；未校准时明确提示仍在用本月基础结余做代理。
- 决策模拟优先使用最新校准现金作为起始安全垫。

验收：

```bash
uv run --with pytest pytest tests/test_cash_balance.py tests/test_web_dashboard.py
```

### Task 7：未来计划事件和实际流水匹配（已完成）

目标：把还没发生但确定性较强的现金流纳入预测，并避免实际发生后重复计算。

要求：

- 支持一次性未来收入 / 支出。
- 支持计划事件启用、停用、已匹配。
- 支持与 BeeCount 实际流水按日期、金额、方向和分类做高置信度匹配。
- matched 计划事件不再进入未来预测。

已落地：

- `planned_cashflow_events` 保存一次性未来收入 / 支出，支持启用、停用、未匹配、已匹配。
- 首页可新增计划事件，并可触发 BeeCount 实际流水匹配。
- 匹配规则按 BeeCount 来源、日期窗口、金额、方向、现金流语义和分类做高置信度匹配。
- 决策模拟纳入未匹配计划事件；已匹配计划事件不再重复计算。

验收：

```bash
uv run --with pytest pytest tests/test_planned_events.py tests/test_match_actuals.py
```

### Task 8：大额消费和买车模拟器（已完成）

目标：回答“大额消费能不能做”。

输入：

- 支出金额。
- 支出时间。
- 是否分期。
- 每月新增固定支出。
- 是否影响收入。

输出：

- 能不能做。
- 做完后安全垫剩几个月。
- 哪个月风险最高。
- 需要等哪笔收入到账。
- 建议金额上限。

已落地：

- 决策模拟支持一次性 / 分期支付。
- 支持录入每月新增固定支出和每月收入影响。
- 模拟结果输出最低现金、安全垫月数、风险月份、建议和建议金额上限。
- Web 表单已暴露新增收入 / 支出影响字段。

### Task 9：投资仓位安全上限（已完成）

目标：先做现金流安全判断，不做投资收益预测。

输出：

- 必须保留现金。
- 可投资现金上限。
- 投资后安全垫变化。
- 当前是否建议暂缓加仓。

已落地：

- 投资模拟不预测收益，只按本金占用现金处理。
- 输出必须保留现金和可投资现金上限。
- 若加仓金额超过现金安全上限，会给出降低金额或暂缓建议。

## 4. 非目标

v0.2 不做：

- 投资收益预测。
- 自动银行连接。
- 多用户、登录、云同步。
- 复杂资产配置模型。
- 完整会计系统。
- AI 黑箱理财建议。

## 5. 开发原则

- 建议先于图表，图表服务于解释。
- 规则先于 AI，结论必须可解释。
- Web 输入和对话输入不能形成两套账。
- 真实账本数据不得提交。
- 每个行为变化必须配 pytest。
