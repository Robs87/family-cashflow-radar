# BeeCount Cloud 数据源适配计划 v0.3

## 1. 目标

BeeCount Cloud 作为家庭流水记录层，Family Cashflow Radar 作为分析和决策层。

新的系统边界：

```text
BeeCount App / BeeCount Cloud
  -> 记录、同步、账户、分类、标签、预算、附件、基础图表

Family Cashflow Radar
  -> 流水镜像、现金流语义、规则分类、月度现金流、预测、建议、决策模拟
```

## 2. 为什么这样分层

BeeCount Cloud 已经提供：

- iOS / Android / Web 三端账本。
- 收入、支出、转账、账户、分类、标签、预算。
- WebSocket 实时同步和离线优先。
- 单容器自托管、SQLite 数据、加密备份。
- MCP 读写工具，能让 LLM 查询交易、记账和改预算。

所以本仓库不应该再投入主要精力做日常记账体验。核心差异化应该是：

- 把普通账本分类翻译成家庭现金流语义。
- 识别债务、投资、工作垫付、报销、资产事件和固定支出。
- 给出可解释的安全垫、风险月份和行动建议。
- 回答“现在能不能提前还贷 / 买车 / 大额消费 / 加仓”。

## 3. 数据接入优先级

### 3.1 MCP 只读验证

先用 BeeCount Cloud MCP 的 read tools 验证字段和语义：

- `list_ledgers`
- `list_transactions`
- `list_accounts`
- `list_categories`
- `list_tags`
- `list_budgets`
- `get_analytics_summary`

用途：快速确认账本结构、日期、金额、交易类型、分类名、账户名和标签能否满足分析。

### 3.2 Read API 同步

稳定实现应优先使用 BeeCount Cloud 的 read API 或 projection 读路径，而不是抓 Web 页面。注意：read API 使用普通 Web/App access token；`bcmcp_...` PAT 只适用于 MCP endpoint，不能直接调用 `/api/v1/read/*`。

同步原则：

- 只读。
- 支持增量同步。
- 支持指定账本。
- 保存 BeeCount 原始 `sync_id` / transaction id。
- 保存完整 raw payload。
- 重复同步幂等。

### 3.3 离线数据库 / 备份导入

BeeCount Cloud 的数据目录和备份可以作为批量导入来源。

适用场景：

- 第一次迁移多年账本。
- 网络不可用时本地分析。
- 灾难恢复后重新构建分析库。

注意：不得提交 BeeCount Cloud 数据库、备份、附件或解密文件。

## 4. 本地数据模型调整

现有表可以复用，但要补齐 BeeCount 来源字段。

建议在 `raw_transactions` 增加或通过 `raw_payload` 稳定保存：

```text
source_system = beecount_cloud
source_ledger_id
source_transaction_id / source_sync_id
source_updated_at
source_deleted_at
source_account_id
source_category_id
source_tags
raw_payload
raw_hash
```

幂等键建议：

```text
beecount_cloud:{ledger_id}:{transaction_sync_id}:{updated_at}
```

如果 BeeCount 交易更新需要覆盖本地镜像，则 raw 层可以保留历史版本，但 normalized 层必须只让最新有效版本参与月度现金流，避免重复计数。

## 5. 同步流水线

推荐 CLI：

```bash
python3 app/scripts/import_beecount.py \
  --db data/processed/cashflow.db \
  --base-url https://bee.332626.xyz:9090 \
  --ledger-id <ledger-id> \
  --access-token-env BEECOUNT_ACCESS_TOKEN

python3 app/scripts/normalize.py --db data/processed/cashflow.db
python3 app/scripts/classify.py --db data/processed/cashflow.db
python3 app/scripts/generate_monthly_cashflow.py --db data/processed/cashflow.db
```

Web 的“刷新数据”后续应改为：

```text
BeeCount sync -> normalize -> classify -> monthly -> advice
```

CSV import 保留为：

```text
historical migration / fallback only
```

## 6. 权限和安全

当前 NAS 地址：

```text
https://bee.332626.xyz:9090
```

鉴权要区分两类 token：

- BeeCount Cloud MCP PAT：形如 `bcmcp_...`，只能访问 `/api/v1/mcp/*`，适合通过 MCP read tools 查询或导出。
- BeeCount Cloud access token：普通 Web/App 登录令牌，可访问 `/api/v1/read/*`，适合本项目的 read API 同步器。

如果本机无法连接该地址，先检查公网反代、NAS 容器端口映射、防火墙、服务监听地址和 Mac/NAS 网络是否互通。

原则：

- Token 只从环境变量或本机私有配置读取，不写入仓库。
- 测试只使用合成 BeeCount payload。
- 不把真实 BeeCount raw payload 写进 fixtures。
- 不默认使用 `mcp:write` 或 write API。
- 如果未来支持写回 BeeCount，必须有显式用户确认、审计记录和幂等冲突策略。

## 7. 验收标准

第一阶段完成时应满足：

1. 能从合成 BeeCount payload 导入交易。
2. 重复导入不重复计数。
3. BeeCount 收入、支出、转账能映射到 `cashflow_direction`。
4. BeeCount 分类、账户、标签能参与规则分类。
5. 工作垫付、报销、投资、债务、资产事件仍按本项目语义处理。
6. 月度现金流和首页建议使用 BeeCount 同步后的数据生成。
7. 真实 BeeCount token、数据库和附件不会进入 git。

## 8. 非目标

v0.3 不做：

- 替代 BeeCount 的移动端记录体验。
- 替代 BeeCount Cloud 同步和备份。
- 抓取 BeeCount Web 页面。
- 多用户权限系统。
- 银行、微信、支付宝自动连接。
- 黑箱 AI 理财建议。
